# 프로젝트 구조 — Feature-Sliced Design (임시 확정)

> 프론트엔드 코드를 어떤 단위로 나누고, 무엇이 무엇을 의존할 수 있는지를 정합니다.
> 방법론은 Feature-Sliced Design(FSD) v2.1을 따르고, Angular에 맞게 매핑합니다. 세부는 구현하며 조정합니다.

## FSD를 쓰는 이유

FSD는 코드를 **레이어 → 슬라이스 → 세그먼트** 세 단계로 나누고, **의존은 아래 방향으로만 흐르도록** 강제하는 구조 방법론입니다. 단일 프론트엔드 코드베이스에서 기능이 늘어날 때, 경계가 무너지지 않게 잡아 줍니다.

핵심 원칙은 **"단순하게 시작하고, 필요할 때 추출한다(Start simple, extract when needed)"** 입니다. 미리 빈 레이어를 만들지 않습니다.

## 레이어 (위 → 아래)

| 레이어 | 역할 |
| ------ | ---- |
| `app` | 앱 초기화, 프로바이더, 라우팅, 전역 스타일 |
| `pages` | 라우트 단위 화면 조합 |
| `widgets` | 여러 페이지가 공유하는 큰 UI 블록 |
| `features` | 재사용되는 사용자 행동 (2곳 이상에서 쓰일 때) |
| `entities` | 도메인 모델 (2곳 이상에서 쓰일 때) |
| `shared` | 비즈니스 로직 없는 인프라 (UI 키트, 유틸, API) |

**임포트 규칙:** 한 모듈은 자기보다 **아래 레이어에서만** 임포트합니다. 같은 레이어의 슬라이스끼리 직접 임포트하는 것은 금지합니다.

## Angular 매핑

Angular의 `src/app`은 부트스트랩·프로바이더·라우터를 이미 담고 있어, 그대로 **FSD의 `app` 레이어**가 됩니다. 나머지 레이어를 `src/` 아래에 형제로 둡니다.

```
src/
  app/                  # FSD app — Angular 루트, providers, 라우터, 전역 스타일 import
    app.config.ts
    app.routes.ts
    app.ts
  pages/                # 라우트 단위 화면
    home/               #   할 일 목록
    notifications/
    settings/
    auth/               #   로그인 · 회원가입
  widgets/              # 여러 페이지가 공유하는 큰 블록 (필요 시)
    app-nav/            #   하단 탭 / 사이드 내비
  features/             # 재사용되는 사용자 행동 (2곳 이상에서 쓰일 때 추출)
    add-todo/
    toggle-todo/
    set-reminder/
    switch-theme/
  entities/             # 도메인 모델 (2곳 이상에서 쓰일 때 추출)
    todo/
    user/
    notification/
    settings/
  shared/               # 비즈니스 로직 없는 인프라
    ui/                 #   디자인 시스템 프리미티브
    lib/                #   breakpoint, gesture, format
    api/                #   Dexie DB, CRUD
    config/             #   라우트 상수 등
  styles.css            # Tailwind + 디자인 토큰(@theme)
  main.ts
```

**세그먼트:** 슬라이스 내부는 `ui/` · `model/` · `api/` · `lib/` · `config/`로 나눕니다. 파일명은 기술 역할이 아니라 **도메인** 기준으로 짓습니다(`todo.ts`, `types.ts` 아님).

**공개 API:** 각 슬라이스는 `index.ts`로만 외부에 노출합니다. `shared`는 슬라이스가 없으므로 세그먼트별로 공개합니다(`shared/ui/index.ts` 등).

**경로 별칭:** `tsconfig.json`의 `compilerOptions.paths`에 `"@/*": ["src/*"]`를 추가해 `@/shared/ui`, `@/entities/todo`처럼 임포트합니다.

## 시작 전략 — 미리 만들지 않습니다

FSD의 권고는 분명합니다. 처음에는 **`app` + `pages` + `shared`만으로 시작**하고, 같은 코드가 *실제로* 두 곳 이상에서 쓰일 때 비로소 `widgets` · `features` · `entities`로 추출합니다.

위 트리의 `features`·`entities` 슬라이스는 **예상되는 추출 지점**일 뿐, 미리 폴더를 만들지 않습니다. `todo`·`user`·`notification`·`settings`는 여러 페이지가 공유할 가능성이 높아 곧 엔티티가 될 것으로 보입니다. 그래도 첫 사용은 페이지 안에 두고, 두 번째 소비자가 생기는 순간 옮깁니다. 경계 검증은 FSD 공식 린터 **Steiger**(`npx steiger src`)로 합니다.

## shared/ui 컴포넌트 인벤토리

> 질문: "요구사항과 상관없는 공용 컴포넌트를 만들 것인가?" → **예.** 도메인을 모르는 프리미티브를 `shared/ui`에 CDK + Tailwind로 만듭니다. 비즈니스 로직은 두지 않습니다(FSD 규칙).

| 프리미티브 | 토대 | 책임 |
| ---------- | ---- | ---- |
| Button · IconButton | Tailwind | 디자인 토큰 버튼 (variants) |
| Sheet | CDK Overlay + BreakpointObserver + FocusTrap | 적응형 바텀시트/모달 |
| Backdrop | CDK Overlay | 딤 배경 |
| TextField · Textarea | Signal Forms 호환 | 텍스트 입력 |
| Checkbox · Switch | Tailwind | 토글 |
| ListItem | CDK (스와이프) | 스와이프 가능한 리스트 행 |
| Snackbar | CDK Overlay + LiveAnnouncer | 토스트 + 되돌리기 |
| Badge · Chip | Tailwind | 태그·라벨 |
| Avatar | Tailwind | 프로필 이미지 |
| Spinner · Skeleton | Tailwind | 로딩 표현 |
| EmptyState | Tailwind | 빈 화면 |
| Nav (TabBar · Rail) | CDK + BreakpointObserver | 적응형 내비게이션 |
| Card | Tailwind | 표면 |

이 프리미티브들은 도메인을 모릅니다. 예를 들어 "할 일 항목"은 `entities/todo/ui/TodoItem`이 `shared/ui`의 `ListItem`·`Checkbox`·`Badge`를 **조합**해 만듭니다. 각 프리미티브의 색·라운드·타이포는 디자인 시스템 문서에서 정의합니다.
